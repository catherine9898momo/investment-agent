对，你理解得很准：**目前还没有真正把真实 LLM 接口跑进主链路里作为核心报告生成器。**

现在代码里有 `AnthropicJSONResearchSynthesizer` 这个入口，也有 `build_synthesis_prompt(run)`，但我们日常验证走的是 `mock` synthesizer。今天做的 `Context Builder` 是在为真实 LLM 接入做地基：以后 LLM 不能随便看整个 run，而是只看 `ResearchContext`。

你问的几个点很关键，我逐个拆开。

**1. LLM 返回了 ContextFact 里没有的 fact_id，能说明它是 Missing Fact 吗？**

不能直接这么说。

如果 LLM 返回了一个不存在的 `fact_id`，它首先说明的是：

```text
这个 claim 当前没有被已知证据支持
```

但它不一定等于 `MissingFact`。

这里有三种可能：

```text
A. LLM 幻觉了一个 fact_id
B. LLM 提到了真实世界里可能存在的事实，但我们当前 context 没有收集到
C. 我们的 normalizer / context builder 漏掉了已有事实
```

所以正确流程不是“fact_id 不存在 -> 直接 MissingFact”，而应该是：

```text
LLM claim
 -> Claim Verifier 检查 fact_id 是否存在
 -> 不存在则标记 unsupported claim
 -> 判断这个 claim 对当前问题是否重要
 -> 如果重要，转成 evidence gap / retrieval need
 -> 再由工具或联网搜索尝试补证据
 -> 找到可靠来源后，变成新的 Fact / VerifiedFact
 -> 找不到，才保留为 MissingFact 或 Unknown
```

也就是说，**MissingFact 是证据缺口，不是 LLM 幻觉本身。**

更严谨地说：

```text
不存在的 fact_id = unsupported claim
当前需要但未收集到的证据 = missing fact
模型编造的事实 = hallucination / invalid claim
```

你说“是不是还要进一步网上搜索官方数据”，是的，尤其是重要 claim。比如 LLM 说：

```text
Micron 下跌主要因为管理层下调毛利率指引。
```

但 context 里没有财报/指引 fact。那不能直接接受，也不能直接说是假的。应该进入：

```text
evidence retrieval
 -> SEC / 公司 IR / earnings transcript / press release / reliable news
 -> normalize
 -> verified fact
 -> 再允许 synthesis 使用
```

这就是后面要做 `Claim Verifier + Retrieval Loop` 的原因。

**2. Attribution Executor 是什么？**

我之前说的 `attribute executer` 更准确应该叫：

```text
Attribution Executor
```

中文可以叫：**归因执行器**。

现在我们有的是 `Attribution Planner`。它只是在说：

```text
如果用户问“为什么跌”，我们需要哪些证据？
```

比如它会规划：

```text
price_move
news_events
sector_move
peer_moves
macro_context
earnings_or_guidance
analyst_actions
```

但 planner 只是“列任务清单”。Executor 是真正去执行这些任务的人。

例如：

```text
sector_move:
  拉 SMH / SOXX / QQQ 等 ETF 同期走势

peer_moves:
  拉 WDC / STX / NVDA / AMD 等同行同期走势

macro_context:
  拉 Nasdaq、10Y yield、美元指数、风险偏好指标

earnings_or_guidance:
  查财报、指引、电话会、公司公告

analyst_actions:
  查评级、目标价调整
```

然后 executor 把这些结果变成 facts：

```text
SMH 同期下跌 2.1%
WDC 同期下跌 3.8%
MU 同期下跌 6.4%
```

这样报告才能说：

```text
这次下跌有板块共振，但 MU 跌幅大于板块，说明可能还有个股因素。
```

没有 executor 的时候，我们只能说：

```text
需要核验 sector_move / peer_moves
```

所以现在是“知道该查什么”，但还没完全“自动查完并归因”。

**3. Research Model Router 是什么？**

现在我们的 `Intent Router` 已经能粗略判断用户在问：

```text
新闻/涨跌原因
估值
持仓
公司研究
直接交易建议边界
```

但这还只是“问题类型路由”。

`Research Model Router` 更进一步，它决定：

```text
应该用哪套研究框架来组织证据和结论？
```

举例：

如果用户问：

```text
美光最近为什么跌？
```

应该选择：

```text
Price Attribution Model
```

它关心：

```text
价格变化
板块/同行
新闻催化
财报/指引
宏观风险偏好
分析师动作
```

如果用户问：

```text
美光现在估值贵不贵？
```

应该选择：

```text
Valuation Review Model
```

它关心：

```text
收入增长
毛利率周期
EPS / FCF
历史估值区间
同行 multiples
周期位置
```

如果用户问：

```text
我组合里 MU 仓位是不是风险太大？
```

应该选择：

```text
Portfolio Risk Model
```

它关心：

```text
仓位比例
相关性
最大回撤
集中度
周期暴露
组合其他持仓
```

你说得对：**目前我们还没有很多真正独立的研究模型。**

所以 Research Model Router 现在还只是未来方向。我们需要先定义几个 model spec，例如：

```text
price_attribution
valuation_review
earnings_quality
portfolio_risk
company_memo
```

每个 model spec 决定：

```text
需要哪些 facts
哪些 facts 是必需
怎么判断 missing
怎么组织 context
报告应该有哪些章节
claim verifier 应该检查哪些风险
```

**4. Claim Verifier 是什么？**

Claim Verifier 就是“模型说完话以后，系统要审它”。

LLM 可能会生成：

```text
MU 下跌主要因为市场担心 DRAM 价格见顶。
```

Claim Verifier 要问：

```text
1. 这句话有没有引用 fact_id？
2. fact_id 是否真实存在？
3. fact_id 的来源是否存在？
4. fact 内容是否真的支持这句话？
5. 这句话有没有把不确定事实说成确定结论？
6. 有没有使用 missing fact 做结论？
7. 有没有包含买入/卖出/加仓/减仓等交易建议？
8. 有没有从“价格下跌”直接跳到“原因是 X”？
```

现在我们已有的是比较基础的：

```text
bind_claims_to_evidence
```

它能检查：

```text
claim 引用的 fact_id 是否存在
fact 是否有 source
source 是否存在
```

但它还不会深度判断：

```text
fact 的文本是否真的支持 claim
claim 语气是否过度确定
claim 是否偷用了 missing fact
claim 是否需要进一步检索
```

所以 Claim Verifier 是 P1 里非常重要的安全层。

**5. 当前最准确的状态**

现在我们已经有：

```text
用户问题 -> 事实收集 -> verified/missing fact table -> research context -> mock synthesis -> evidence binding -> memo
```

但真实 P1 还缺：

```text
真实 LLM synthesis
Claim Verifier
unsupported claim -> retrieval need 的闭环
Attribution Executor
Research Model Router / model spec
```

我建议下一步不要急着接真实 LLM。更稳的顺序是：

```text
1. 先做 Claim Verifier 的最小版本
2. 让它能区分 unsupported claim / missing fact / invalid hallucination
3. 再接真实 LLM
4. 最后做 attribution executor 补真实归因数据
```

这样真实 LLM 接进来时，我们不是“相信它”，而是“审计它”。这对投资研究系统很关键。
