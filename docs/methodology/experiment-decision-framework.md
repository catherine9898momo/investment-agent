# 实验 vs 推理：何时真做、何时记录推断

## 适用场景

学习路线 / 工程验证 / 简历素材准备过程中，频繁会遇到一类决策：**"这个实验我到底要不要花时间真做？"**

错误的默认值是"按计划执行 → 做"。这会浪费稀缺时间在低增量任务上，让真正稀缺的实验（如上下文压缩、跨会话记忆）没有 buffer。

本文给出**决策框架**，每次准备开实验前用 1 分钟过一遍，决定 "真做 / 推理记录 / 完全 skip" 三档。

---

## 决策维度：三个信号

每个实验沿三个维度打分（高 / 中 / 低）：

### 维度 1：先验确定度

> "如果我不做这个实验，结论可不可靠地推断出来？"

| 等级 | 信号 | 例子 |
|---|---|---|
| **高（结论已确定）** | 我能列出实验所有可能结果，且置信度 > 90% | "移除 `required` → LLM 会漏传字段或脑补"——TS 工程师常识 |
| **中（方向已知，数字未知）** | 我知道 v0 大概率比 v1 差，但差多少不知道 | "v0 description 在陷阱 case 上失败率多少" |
| **低（结论可能反直觉）** | 我对结果完全没把握，甚至可能验证自己错了 | "三种上下文压缩策略哪种 token 节省最多 / 质量损失最少" |

### 维度 2：学习增量

> "做完之后，我的知识地图会新增一个之前不会写的概念吗？"

| 等级 | 信号 |
|---|---|
| **高** | 实验过程中要解决一个全新的工程难题（如何衡量"答案质量"？怎么设计 LLM-as-judge？） |
| **中** | 实验主要是验证已学概念在新场景下的成立性 |
| **低** | 实验是机械重复"我已经在类似场景做过"的结构（如再做一次 description A/B） |

### 维度 3：简历叙事增量

> "做完之后，我能在面试中讲出之前讲不出的新故事吗？"

| 等级 | 信号 |
|---|---|
| **高** | 是一个可量化、可对比、能讲 trade-off 的全新故事（如"截断 60%/质量降 15% vs 摘要 40%/质量降 3%"） |
| **中** | 给已有故事增加 1-2 个新数据点 |
| **低** | 与已有故事高度重复，加进去顶多多 1 句话 |

---

## 决策矩阵

按三维打分后查表：

| 先验确定度 | 学习增量 | 简历增量 | 决策 |
|---|---|---|---|
| 低 | 高 | 高 | ✅ **真做**（核心实验，全力投入） |
| 低 | 高 | 中 | ✅ **真做**（学习优先） |
| 中 | 高 | 中 | ✅ **真做**（值得） |
| 中 | 中 | 高 | ✅ **真做**（叙事优先） |
| 高 | 中 | 高 | 🟡 **小型实验**（只跑足够拿数字的最小 case 集） |
| 高 | 低 | 高 | 🟡 **推理 + 文档化**（写下结论 + 决策原因即可，保留叙事） |
| 高 | 低 | 中 | ❌ **skip**（写在某处一句话过掉） |
| 高 | 低 | 低 | ❌ **完全 skip**（连记都不用记） |

**Rule of thumb**：三个维度同时落到中位或以上时再做；任意一个维度 "高" 就要慎重 skip；两个维度 "低" 直接 skip。

---

## 关键反例：什么时候**必须做**

即使先验确定度高，也有必须做的情况：

1. **结论是"原理上 X，但工程上 ?"**——原理已知不等于工程已验证。比如"description 边界写法重要"是原理，但"在 Claude Desktop 含 ToolSearch retrieval 层的真实环境里效果如何"是工程问题
2. **简历需要"做过 X"作为可核查事实**——面试官追问"你说做过 A/B 实验，case 几个？数据呢？"时，没数据无法 hold 住
3. **决策路径上的关键变量未知**——如果接下来的设计取决于这个实验结果（"如果 v0 在 X 上失败，我会改 Y"），必须做

---

## 关键反例：什么时候**绝对不该做**

1. **同一个原理验证过两次以上**——再做就是仪式感
2. **预期 ROI < 已确定低 ROI 任务**——比如花一晚做低增量实验 vs 花一晚啃下 W5 全新内容
3. **基础设施还没到位时强行做**——如果 API 502 / yfinance 挂了 / 工具链没 ready，先解决卡点再说

## 应用案例 A：description A/B（2026-05-12）

| 维度 | 评分 | 备注 |
|---|---|---|
| 先验确定度 | **中** | 4 月 my-cli-agent 做过类似实验，方向已知，但**真实业务工具 + 含陷阱 case 集 + Claude Desktop 环境**是新组合，具体表现未必能精确预测 |
| 学习增量 | **中** | description 工程价值结论已沉淀，但"含陷阱的 case 设计"+"Desktop ToolSearch 平台层"是新增量 |
| 简历叙事增量 | **高** | 真实业务陷阱 case 比 4 月 demo 强，是面试素材的硬升级 |

**决策**：v1 baseline 跑全 + v0 量化跑 skip（后者落到"高确定度 + 低学习 + 中叙事增量" → 推理+记录）。

事实结果：v1 baseline 10/10 + Case 3/4/5 表现，简历素材已饱和，确认 skip 决策正确。

## 应用案例 B：inputSchema A/B（2026-05-13）

| 维度 | 评分 | 备注 |
|---|---|---|
| 先验确定度 | **高** | `required` / `enum` / `type` 三个字段作用是 TS 工程师常识，失败模式可精确推断 |
| 学习增量 | **低** | 不会带来任何"以前不会的新概念" |
| 简历叙事增量 | **低** | 与 description A/B 故事重复度 70%，加进去顶多多一句话 |

**决策**：完全 skip 实测，**用推理写一份"预期失败模式 + 决策原因"文档**保留路线痕迹和判断依据，节省的时间投到 finance_server / W5。

## 元判断：什么时候这个框架本身要被修正

不要把框架当圣经。在以下信号出现时，框架要被重新审视：

1. **连续 3 次 skip 决策被后续证明错了**——可能"先验确定度"评估偏乐观
2. **三维评分全部"高"但还是做错方向了**——可能维度不全，需要补"基础设施/时间窗口"等维度
3. **始终在做高确定度 + 中叙事的"安全实验"**——可能在回避"低先验确定度"的真核心实验，需要刻意 push 出舒适区
