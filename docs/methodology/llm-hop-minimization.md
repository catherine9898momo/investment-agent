# LLM Hop 最小化原则 ——「需要保真 vs 需要判断」的设计分水岭

> 这一原则是用户在 5/14 反例闭环实测后，**自发推导**出来的。原句作为本文档主体，工程术语对照只做辅助。

---

## 用户原句（不重写，作为本文档核心）

### 失败机制（why）

> 这种失败模式就是 LLM 处理非结构化自然语言由于**注意力机制**，天然存在对关键事实数据产生**忽略、压缩等处理偏移**的可能性的本质原因。

### 设计原则（how）

> 需要 X 的信息，应该是**不能被忽略的具体事实，不会被 LLM 多次处理**；
>
> 需要 Y 的判断，应该自然语言段落，提供 LLM 做**意图识别和判断**，内容可以被多次处理且**省略、压缩、丢失不会影响预期处理结果的输出**。

---

## 工程术语对照

| 用户原话 | 工程术语 | 含义 |
|---|---|---|
| "不能被忽略的具体事实" | High-fidelity factual lookup | 高保真事实查询——容错率为 0 |
| **"不会被 LLM 多次处理"** | **LLM hop 最小化** | 每经过一次 LLM 解析 = 一次注意力损失风险 |
| "注意力机制 / 处理偏移" | Attention drift / compression loss | 注意力机制的机制层归因（不是"模型不够大"，是机制本身的特性）|
| "意图识别和判断" | Pattern recognition / intent matching | LLM 的优势区——容错率高 |
| "丢失不会影响预期输出" | Fault-tolerant context | 容损上下文——LLM 处理这种输入很 OK |

---

## 判断流程（基于这条原则做工具/数据流设计时）

```
设计某个工具的输入/输出时，先问两个问题：

1. 这条信息的"保真度要求"是什么？
   - 关键事实（日期/比例/金额/ID）→ 必须高保真 → 走结构化工具
   - 模糊判断（"看起来异常"、"利好利空"）→ 容损 OK → LLM 自由处理

2. 这条信息会被 LLM 处理几次（hop 几次）？
   - 每多一次 hop = 一次"压缩/省略/偏移"的风险
   - 关键事实应该 hop 次数 = 0（直接给 Claude 已解析好的字段）
   - 模糊判断可以 hop 多次（LLM 越想越好）
```

## 应用 1：corporate_actions_server（5/14 实例）

**问题场景**：Claude 需要算 TSLA $1200 复权后成本。

**错误设计**（5/13 实际反例）：
- Claude 直接用训练知识里"模糊记得的 TSLA 拆股事实" → hop=∞（每次回答都重新从注意力分布里采样）
- 失败模式：早上对（15:1），晚上错（1:3，漏 2020）

**正确设计**（5/14 修复）：
- `get_corporate_actions("TSLA")` → 返回 `[{date:'2020-08-31', factor:5.0}, {date:'2022-08-25', factor:3.0}]`
- LLM hop = 0（拿到的就是已解析的结构化字段，直接乘）
- 失败模式：消除（除非工具本身坏）

**为什么 web_search 不行**：
- `web_search("TSLA stock split")` → 返回 10 条新闻段落
- Claude 拿到后必须解析 → hop=1（在 Claude 内部多了一次"读自然语言抽取数字"的步骤）
- 这一步就会触发"注意力机制的忽略 / 压缩 / 偏移"

## 应用 2：触发器 vs 决策 vs 事实（与 architecture.md 5/13 节的关系）

5/13 architecture.md 写了"知识分层"模式（触发器 / 具体事实 / 决策建议三层）。**本原则是它的机制层归因 + 通用化**：

- 触发器（"价位看起来不对"）→ LLM 模式识别 → **fault-tolerant，hop 可以 >0**
- 具体事实（"$1200 / 15 = $80"）→ 结构化工具 → **必须 hop=0**
- 决策建议（"持有 70-80%"）→ LLM 综合推理 → **fault-tolerant，hop 可以 >0**

5/13 是**模式描述**（要分三层）；本原则是**判断标准**（怎么判断哪一层）+ **机制归因**（为什么要分层）。

## 反例：什么时候 OK 让 LLM 处理非结构化？

不是所有信息都要"hop=0"。明确容损的场景：

| 场景 | 为什么 OK | 例子 |
|---|---|---|
| 用户意图识别 | 容错率高，错了用户会纠正 | "我想看持仓" / "查一下苹果" |
| 多新闻源摘要 | 信息冗余度高，漏一条不影响判断 | 给定 10 条新闻让 LLM 提炼利好利空 |
| 自然语言生成 | 输出本身就是"概括化" | 投资建议 / 周报正文 |
| 边界判断 | 阈值不需要精确 | "这价位是不是异常" |

---

## 反例：什么时候必须 hop=0？

| 场景 | 为什么必须 | 例子 |
|---|---|---|
| 金额 / 价格 / 比例 | 错一个数字业务全错 | 拆股因子、汇率、复权后成本 |
| 日期 / 时间 | 用作过滤条件，错了拉错数据 | "查上周新闻" → 日期必须准 |
| ID / 编码 | 错一位拉错记录 | 股票代码、订单号 |
| 法律 / 合规事实 | 错了承担法律责任 | KYC 记录、监管文件引用 |

---

## 简历叙事价值

5/13 反例 + 5/14 闭环的故事，**机制层归因这一层**（"注意力机制 → 处理偏移"）很多 P5/P6 候选人答不上来。能脱口而出"注意力机制天然存在压缩偏移，所以工程上要 hop 最小化"——这是 P6+ 工程师对 LLM 失败模式的工程认知证据。

面试场景下能用这一条原则：
1. 解释"为什么垂直 Agent 一定要有 ground truth 工具"
2. 解释"什么时候用 RAG，什么时候用结构化数据库"（RAG = hop ≥ 1，不适合必须保真的场景）
3. 解释"工具设计的 description / inputSchema / 返回结构三件套"——每一件都是在减少 LLM hop 路径上的丢失风险
