# 复杂目标防漂移：从“连续 yes”到可审计执行闭环

> 触发场景：复杂工程目标在多轮 LLM 协作中不断被局部任务稀释。用户持续确认“yes”后，模型容易忘记原始目标、早期约束和自己提出过的规划，最终做完很多局部正确的事，但整体方向偏移。

---

## 1. 问题定义

长任务里的主要风险不是模型不会做单步任务，而是它会在多轮执行中发生目标漂移：

- **目标遗忘**：原始目标被后续局部指令覆盖，模型只记得最近几轮。
- **规划蒸发**：早期提出的阶段性计划没有成为可执行状态，只存在于对话文本里。
- **确认偏置**：用户连续说 yes 时，模型会把“继续”理解成“当前方向完全正确”，很少主动回到总目标校验。
- **局部最优累积**：每一步看起来都合理，但组合起来没有完成最初想要的复杂系统建设。

这类问题本质上是上下文工程问题，也是执行治理问题：不能只依赖模型“记得”，必须把目标、计划、约束和验收标准外化成可检查状态。

---

## 2. 工程原则

### 原则 A：目标必须外化，不只存在于聊天上下文

把复杂目标写成稳定资产，而不是靠对话记忆维持。

推荐形态：

- `Goal Brief`：一句话目标、非目标、成功标准、关键约束。
- `Execution Plan`：阶段、任务、依赖、验收条件。
- `Decision Log`：为什么选这个方向、放弃了哪些替代方案。
- `Open Questions`：当前不确定事项，不让模型假装已经确定。

对应到本项目，可以落在 `docs/PROJECT_PANORAMA_AND_MILESTONES*.md`、`docs/EXECUTION_PLAN_P1.md`、`docs/methodology/` 和 showcase ADR 中。

### 原则 B：计划要变成状态，而不是一次性叙述

规划如果只是一段自然语言，很容易在后续轮次里消失。更稳的做法是维护显式状态：

- 当前阶段是什么。
- 哪些任务已完成、进行中、未开始。
- 哪些建议已采纳、推迟或废弃。
- 每次变更是否影响原始目标。

对于多日任务，状态应该能被重新加载，而不是只能从完整聊天记录里恢复。

### 原则 C：每个局部任务都要回链到总目标

执行前问一句：

> 这一步服务哪个目标？完成后如何证明没有走偏？

如果回答不上来，这一步很可能只是“看起来有用”。复杂系统里最容易消耗时间的，往往是这些没有回链的局部优化。

### 原则 D：定期做漂移审计

长任务需要周期性检查，不等到最后才验收。

推荐节奏：

- 每 3-5 个子任务检查一次目标一致性。
- 每次上下文压缩、恢复、切线程后先重读 Goal Brief。
- 每个里程碑结束时做一次“原始目标 vs 当前产物”的差异表。

审计不是复盘形式主义，而是防止“越做越像另一个项目”。

## 3. 可复用流程

### Step 1：写 Goal Brief

最小模板：

```md
## Goal Brief

目标：
非目标：
成功标准：
关键约束：
当前里程碑：
风险：
```

### Step 2：把计划拆成可验收任务

每个任务至少包含：

- 产物路径或行为变化。
- 验收方式。
- 依赖项。
- 不做什么。

### Step 3：执行中维护状态

建议每轮结束时更新：

- `Done`
- `Next`
- `Changed Decisions`
- `Risks / Drift Signals`

### Step 4：漂移审计

用四个问题做检查：

| 检查项 | 问题 |
|---|---|
| Goal | 当前工作是否仍服务原始目标？ |
| Plan | 早期规划里还有哪些未完成或已不适用？ |
| Evidence | 有哪些文件、测试、实验结果能证明进展？ |
| Drift | 最近新增任务是否改变了范围、优先级或验收标准？ |

### Step 5：把偏移变成新决策

发现偏移不一定要回退。有些偏移是合理学习后的方向调整。关键是把它显式记录成 decision，而不是悄悄漂走。

## 4. 面试表达

30 秒版：

> 我后来意识到，复杂 Agent 项目里 LLM 最大的问题不只是单轮幻觉，而是多轮执行的目标漂移。用户一直说 yes 时，模型会越来越关注最近的局部任务，忘掉原始目标、早期规划和验收标准。所以我把目标、计划、决策和验收外化成稳定文档，并在里程碑和上下文切换时做漂移审计。这个思路和我项目里的 Source / Fact / Claim / Trace 是同一类工程化约束：不要让关键状态只存在于模型上下文里。

追问版：

> 我的判断是，长任务治理不能靠“让模型更努力记住”，而要把目标状态产品化。Goal Brief 定义方向，Execution Plan 定义步骤，Decision Log 记录取舍，Trace / Eval 证明产物有没有满足目标。模型可以负责执行和归纳，但关键控制面要外化、可恢复、可审计。

---

## 5. 与本项目的关系

这条方法论和现有三条主线互相补强：

- **知识分层**：关键事实不能只靠 LLM 记忆，要落到结构化 ground truth。
- **LLM hop 最小化**：关键数字、日期、状态尽量 hop=0，避免自然语言中转造成偏差。
- **Production Research Loop**：Source / Fact / Claim / Evidence / Trace 把研究结论变成可审计状态。

复杂目标防漂移是同一个思想在项目管理和 Agent 执行层的应用：关键状态外化，模型负责推理，系统负责约束与追踪。
