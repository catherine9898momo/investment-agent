## 钩子段（opener）

> 在做 Agent 项目时，我们经常被问到一个问题：**这件事让 LLM 做合适，还是让工具做合适？**
>
> 这个问题没有标准答案 —— 因为它问错了维度。
>
> 真正应该问的是两件事：**这条信息的保真度要求是什么？这条信息会被 LLM 处理几次？**
>
> 这篇文章提出一套基于这两个维度的设计模式 —— **触发器 / 具体事实 / 决策建议** 三层知识分层。它在我自己的金融 Agent 项目里被一个反例反向逼出来，但抽象之后适用于任何"LLM + 工具"的设计场景。
>
> 这套模式的核心原则只有两句话：**LLM 提示要存在感、不要权威感；LLM hop 越少，关键事实越保真**。

## 大纲

### Part 1 · 为什么"让 LLM 还是让工具"问错了维度（约 500 字）

**核心论点**：传统的"LLM vs 工具"二元划分太粗。

要点：
- 误区 1："简单任务给 LLM、复杂任务给工具" —— 颠倒了
- 误区 2："工具 > LLM，能用工具就用工具" —— 失去 LLM 推理价值
- 误区 3："给 LLM 联网搜就能解决幻觉" —— 没解决
- 真问题：**信息的保真度要求** + **LLM hop 次数**
- 引出三层模式

### Part 2 · 反例引入 — 我是怎么撞到这个问题的（约 600 字）

**核心论点**：模式不是设计出来的，是反例逼出来的。

要点：
- 简述 TSLA 反例（同一天两次 session 给出 15:1 vs 3:1）
- 复权后成本 $80 vs $400，浮盈方向反转
- 反例发现的运气 + 系统性反思
- 这个反例让我意识到：**之前在工具设计时，没有显式区分"保真信息"和"判断信息"**
- 设计反推：什么样的设计原则可以预防这类反例？

**配图**：02-knowledge-layering 的反例失败模式分图

### Part 3 · 三层模式定义（约 800 字）

**核心论点**：三层各司其职，决策标准明确。

| 层 | 角色 | 提供者 | 容错率 | LLM hop | 例子 |
|---|---|---|---|---|---|
| 1. 触发器 | "可能有问题" | LLM 训练知识 | 高 | 可 >0 | "$1200 看起来比现价高" |
| 2. 具体事实 | 日期/比例/数字 | 外部 ground truth | **必须 0** | **必须 0** | TSLA 拆股 2020-08 5:1 |
| 3. 决策建议 | 持有/加仓/减仓 | LLM 综合 | 高 | 可 >0 | "建议持有 70%-80% 仓位" |

要点：
- 每一层的角色定位
- 容错率 vs hop 次数的两维矩阵
- 第 2 层为什么"必须 hop=0"
- 三层之间的数据流（触发器 → 事实 → 建议）
- 不是层级关系（不是 1>2>3），是分工关系

**配图**：02-knowledge-layering 主图

### Part 4 · 机制层归因 — 为什么 hop 这件事重要（约 700 字，最深的一段）

**核心论点**：LLM 注意力机制天然存在压缩偏移。

要点：
- 用户原句（我自己提炼）："LLM 处理非结构化自然语言由于**注意力机制**，天然存在对关键事实数据产生**忽略、压缩等处理偏移**的可能性"
- 这一原则的工程术语：**LLM hop 最小化（LLM hop minimization）**
- 每多一次 hop = 一次"压缩/省略/偏移"的风险
- 反例对照：
  - hop=0：`get_corporate_actions("TSLA")` → 返回 `{cumulative_factor: 15.0}` → 直接乘
  - hop=1：`web_search("TSLA stock split")` → 返回 10 条新闻段落 → Claude 读再抽数字 → 偏移
- 为什么 RAG 不适合"必须保真"场景：RAG = hop ≥ 1
- 为什么结构化数据库适合：返回字段就是已解析结果，hop = 0

### Part 5 · 应用案例 — 不止金融（约 600 字）

**核心论点**：模式可迁移到任何垂直 Agent。

要点：
- **金融**（本项目）：拆股因子、汇率、复权后成本
- **医疗**：药物剂量、过敏史、化验结果数值
- **电商**：库存数、SKU 编码、订单金额
- **客服**：用户 ID、订单号、合约条款引用
- **企业 RAG 改造**：哪些 chunk 该保留为"事实"层（直接结构化）、哪些走 LLM 摘要
- 通用判别清单：
  - 错一个数字业务全错？→ 第 2 层
  - 过滤条件？→ 第 2 层
  - 法律/合规事实？→ 第 2 层
  - 意图识别？→ 第 1 层
  - 多源摘要？→ 第 3 层
  - 自然语言生成？→ 第 3 层

### Part 6 · 核心原则 — 存在感不要权威感（约 500 字）

**核心论点**：这个原则比模式本身更重要。

要点：
- 反面例子：Claude 说"复权后成本是 $80 USD，浮盈 +361%" → 用户当真
- 正面例子：Claude 说"可能是拆股前价位，建议核对" → 触发用户主动确认
- UX 层面的体现：
  - Case D' 涌现行为 2：Claude 主动弹 4 选项让用户澄清持仓时点
  - 这是"存在感"的真实落地
- 为什么这个原则反直觉：开发者倾向让 AI"看起来很聪明"
- 但生产系统的"看起来聪明" = "悄悄出错" = "用户失去信任"
- 设计 Agent 的伦理底线

### Part 7 · 与其他设计模式的关系（约 500 字）

**核心论点**：三层模式不孤立，与现有设计模式有对应关系。

要点：
- 与 **RAG**：RAG 的 retrieve+rerank+generate 三步对应"触发 → 召回 → 综合"，但 retrieve 的 chunk 是 hop=1
- 与 **Tool Use**：Function Calling 是三层模式的具体实现手段
- 与 **Multi-Agent**：W7 LangGraph Supervisor-Worker 模式可以将三层映射到不同 Agent
- 与 **CoT / ReAct**：CoT 是第 3 层（决策建议）的展开形式，与本模式不冲突
- 与 **DSPy / TextGrad**：自动 prompt 优化在第 1、3 层有空间，第 2 层不适用

### Part 8 · 局限与未解问题（约 400 字）

**核心论点**：诚实呈现 —— 不是银弹。

要点：
- 边界模糊 case：有些信息介于"事实"和"判断"之间（"这家公司财报好吗" —— 好坏是判断、具体 EPS 是事实），需要拆开
- ground truth 工具坏了怎么办：stale fallback 是兜底但不是根治
- 工具池大了 description 怎么不互相干扰：未解
- 多个 ground truth 来源冲突时怎么仲裁：未解（值得未来研究）

### 结尾 · 一句话总结（约 200 字）

> 真正区分"该让 LLM 做"还是"该让工具做"的标准，不是任务难度，是**保真度要求**与 **LLM hop 次数**。把它分成触发器 / 事实 / 建议三层，把第 2 层强制 hop=0，把第 1、3 层留给 LLM —— 这是我从一个反例里抽象出来的、可被任何"LLM + 工具"项目复用的设计模式。
