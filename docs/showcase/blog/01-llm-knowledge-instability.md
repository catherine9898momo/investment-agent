## 钩子段（opener · 直接复制使用）

> 上线第二天我撞到一个让我后背发凉的现象 —— 同一份持仓数据、同一个 Agent、同一天，Claude 给我了两个不同的答案。
>
> 早上问"我的 TSLA 浮盈多少"，Claude 完整识别出 TSLA 历史上两次拆股（2020-08 5:1 + 2022-08 3:1 累计 15:1），复权后成本 $80，浮盈方向正确。当晚再问"持仓 TSLA + 新闻 + 建议"，Claude 只识别出 2022 年那次 3:1，复权后成本变成 $400，浮盈方向直接反转。
>
> 如果不是我自己撞到，我根本不会怀疑 LLM 的答案 —— 它给得"权威感"十足。
>
> 这篇文章记录我从发现反例到落地修复的 24 小时全过程，以及由此抽象出的一套设计模式：**触发器 / 具体事实 / 决策建议** 三层知识分层。
>
> 如果你也在做 Agent，特别是垂直领域 Agent，这个反例可能值得你警惕。

## 大纲

### Part 1 · 反例发生（约 600 字）

**核心论点**：LLM 训练知识在长 session、多工具调用、输出长建议的上下文压力下，会被"压缩"。

要点：
- 项目背景一句话（金融垂直 MCP Agent，3 Server 协作）
- 反例对比表（早 vs 晚两次回答的拆股识别 / 复权计算 / 浮盈方向）
- 视觉关键点：**1200 ÷ 15 = 80** vs **1200 ÷ 3 = 400**
- 关键追问："谁对？"—— 实际 TSLA 真的有两次拆股，**早上对、晚上漏**
- 严重性：用户没法判断哪个对、Agent 没有"我不确定"信号

**配图**：架构图 02-knowledge-layering 的反例失败模式分图

### Part 2 · 错误归因 vs 正确归因（约 700 字）

**核心论点**：这不是"训练数据过时"，是注意力机制本身的特性。

要点：
- 错误归因 1："Claude 训练数据没更新到 2024" → 反驳：2020-08 是 5 年前事件，更早的
- 错误归因 2："你应该用更新的模型" → 反驳：换 Opus 4.7 也复现
- 错误归因 3："prompt 写得不够好" → 反驳：早上 prompt 更短，晚上 prompt 更详细
- **正确归因**：LLM 处理非结构化自然语言由于注意力机制，天然存在对关键事实数据**忽略、压缩等处理偏移**的可能性
- 验证：早上 Case 5 上下文短、任务专一（算浮盈）；晚上 Case 4 一次串调 4 个工具 + 输出长建议，更易"概括化"
- 这是机制层归因 —— 任何 LLM 在足够压力下都会出现

**配图**：可画一张"上下文压力 → 注意力压缩 → 关键事实丢失"的流程示意

### Part 3 · 工程修复方案选型（约 800 字）

**核心论点**：必须有外部 ground truth，但选哪种 ground truth 关键看 **LLM hop 次数**。

要点：
- 候选方案 4 个（决策树展开）：
  - A. 让 LLM 联网搜（`web_search`）
  - B. 手维护静态 JSON
  - C. yfinance + SQLite 缓存
  - D. 自建数据爬虫
- 关键判别维度：**LLM hop 次数**
  - 联网搜：hop=1（LLM 读 10 条新闻段落再抽数字）→ 失败模式仍存在
  - SQLite 缓存：hop=0（结构化字段直给）→ 失败模式消除
- 其他维度：实施成本 / 实时性 / 合规风险 / 24h 闭环可行性
- **选 C 的关键理由**：hop=0 + yfinance 权威 + 缓存性能 + stale fallback
- 实施细节：300 行代码、24h TTL、复用 memory.db PK (symbol, action_type, date)

**配图**：ADR-003 的决策树图

### Part 4 · 验证与涌现行为（约 600 字）

**核心论点**：修复后不仅 4/4 维度全过，还出现 3 个未设计的涌现行为。

要点：
- Case D' 5/14 实测：
  - 工具调用 4 → 7
  - 复权后成本 $400 → $80
  - 浮盈方向 -64% → +457%
- 涌现行为 1：**金句逐字命中** —— Claude 自发输出"价位异常 → 先查历史拆股事实"，与设计文档措辞一致
- 涌现行为 2：**主动澄清持仓时点** —— Claude 弹 4 选项让用户确认，体现"存在感不要权威感"
- 涌现行为 3：**跨 Server 协同提议** —— Claude 建议更新 portfolio 口径
- 涌现的本质：**description nudge 的溢出效应**，不是 LLM 随机

**配图**：03-case-d-sequence 时序图

### Part 5 · 抽象成模式（约 700 字）

**核心论点**：从这个反例可以抽象出 **三层知识分层** 通用设计模式。

要点：
- 第 1 层 **触发器**："这里可能有问题" → LLM 训练知识 → 容损 OK，hop 可 >0
- 第 2 层 **具体事实**：日期 / 比例 / 数字 → 外部 ground truth → **必须 hop=0**
- 第 3 层 **决策建议**：持有 / 加仓 / 减仓 → LLM 综合 → 容损 OK，hop 可 >0
- 核心原则：**LLM 提示要存在感、不要权威感**
- 应用迁移：W6 实体记忆系统、企业 RAG、医疗 Agent 都可套用
- 与 RAG 的关系：RAG 召回的 chunk 是 hop ≥ 1（LLM 要读再抽），所以 RAG 不适合"必须保真"的场景

**配图**：02-knowledge-layering 三层模式图

### Part 6 · 反思与教训（约 500 字）

**核心论点**：发现反例的运气 + 主动修复的态度。

要点：
- 反例是怎么发现的：同一天两次同问句，被我撞到。**这是个体经验**，不是系统验证
- 工程上的反思：之后每个 case 都跑 2-3 次 session 看稳定性 → 简历素材的可信度更高
- 学习方法的反思：写完不等于内化（5/13 W2 KR 验收完晚上才发现的，证明"做完即遗忘"风险真实存在）
- 提议同行：你的 Agent 项目里，哪些信息现在是 hop>0 的？应该 hop=0 吗？
