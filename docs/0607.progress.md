可以这样理解：**我们现在已经有了“链条骨架”，但还没有完整的“研究 agent 闭环”。**

你描述的目标链条是：

```text
用户投资问题
 -> 理解问题
 -> 选择公司 / 标的
 -> 选择研究模型 / 归因模型
 -> 规划需要哪些数据
 -> 执行工具拉数据
 -> 事实核验 / 缺失检测
 -> context builder 组织上下文
 -> 调 LLM
 -> 生成用户可读研究报告
```

现在每一段的状态如下。

**1. 问题理解：部分具备**

已有：

- `QueryIntake`
- `IntentRoute`
- `ResolvedEntity`
- `TimeWindow`
- `AttributionPlan`

可以识别：

- 用户是不是问新闻/涨跌原因
- 是不是问估值
- 是不是问持仓/组合
- 是不是直接问买卖/加减仓
- 美光 / Micron / MU 这类别名
- 最近、大跌、下跌这类时间/事件表达

缺失：

- 复杂多意图拆解
  ```text
  美光为什么跌？顺便看看现在估值贵不贵，还适不适合继续跟踪
  ```
- 多标的问题
  ```text
  MU 和 NVDA 哪个更值得研究？
  ```
- 更完整的中文公司名 / ticker resolver
- 用户历史上下文继承
  ```text
  那它和上次我们看的英伟达比呢？
  ```

**2. 选择公司 / 标的：部分具备**

已有：

- 规则别名字典
- ticker override
- 中英文别名：
  - 美光 / Micron / MU
  - 特斯拉 / Tesla / TSLA
  - 英伟达 / Nvidia / NVDA
  - 腾讯 / 0700.HK
  - 阿里 / BABA / 09988.HK

缺失：

- 大规模实体库
- 模糊匹配
- 多市场同名公司 disambiguation
- 用户持仓/关注列表辅助解析
- 自动确认低置信度标的

**3. 选择研究模型 / 归因模型：只有 P0 骨架**

已有：

- `AttributionPlan`
- 可识别：
  - price_drop
  - price_rise
  - general
- 对“为什么跌”会规划：
  - price_move
  - news_events
  - corporate_actions
  - sector_move
  - peer_moves
  - macro_context
  - earnings_or_guidance
  - analyst_actions

缺失：

- 真正的模型选择器
  ```text
  估值模型
  财报解读模型
  行业周期模型
  下跌归因模型
  持仓风险模型
  thesis monitoring 模型
  ```
- 不同模型的输入/输出 schema
- 多模型组合
- 模型执行后的统一 synthesis contract

现在其实是：

```text
如果是涨跌问题 -> 用 attribution plan
否则 -> 普通 research plan
```

还不是成熟的“研究模型路由”。

**4. 规划需要哪些数据：部分具备**

已有：

- `ResearchPlan`
- `AttributionPlan`
- `FactNeed`
- `AttributionNeed`
- `MissingFact`

可以列出“需要什么”：

```text
price
history
news
corporate_actions
sector_move
peer_moves
macro_context
earnings_or_guidance
analyst_actions
```

缺失：

- 根据问题动态调整数据粒度
  - 单日跌幅
  - 区间跌幅
  - 最大回撤
  - 事件日
  - 财报窗口
- 计划到具体工具参数
  ```text
  get_history(MU, start=..., end=...)
  get_history(SOXX, same window)
  get_news("Micron shares tumble", days=7)
  ```
- 工具调用顺序/依赖关系
- 如果第一轮证据不足，自动二次补查

**5. 执行工具拉数据：部分具备**

已有工具：

- `finance.get_quote`
- `finance.get_history`
- `news.get_news`
- `corporate_actions.get_corporate_actions`
- `memory.get_preferences`

已有 live-first：

- 默认尝试 live
- live 失败时生成 failure fact
- 不会用 fixture 假装真实数据

缺失：

- 批量同行/指数拉取
  - MU
  - NVDA
  - AMD
  - AVGO
  - QQQ
  - SMH
  - SOXX
- 财报/指引数据
- 分析师评级/目标价
- SEC filing / earnings transcript
- 宏观数据
- 新闻多 query 策略
- 新闻来源质量排序
- 工具结果去重与聚合

最关键缺口是：

```text
sector_move 和 peer_moves 现在只是计划项，尚未真正执行工具拉数据。
```

**6. 事实核验 / 缺失检测：P0 已有，但还弱**

已有：

- `VerifiedFact`
- `MissingFact`
- `build_verified_fact_table`
- 报告里会显示：
  ```text
  已形成 x 条可追踪事实，另有 x 项证据缺口
  ```

已有能力：

- 有工具事实 -> verified fact
- 归因计划里没覆盖 -> missing fact
- 不会假装缺失事实已经存在

缺失：

- claim-level verifier
  ```text
  报告中的每一句关键归因是否有对应事实支持？
  ```
- 多源交叉验证
- 冲突事实处理
  ```text
  新闻说跌，价格数据说涨
  ```
- 来源可信度评分
- fact freshness 策略
- “强事实 / 弱事实 / 假设”分层
- LLM 输出后的二次校验

现在是：

```text
事实表能告诉你缺什么
```

还不是：

```text
每个结论都被严格验证过
```

**7. Context Builder：基本缺失**

这是你刚才提到的链条里非常关键的一环。当前还没有真正的 context builder。

现在的流程大致是：

```text
facts -> mock synthesizer / renderer
```

缺少：

```text
context_builder.py
```

它应该负责把这些东西组织成 LLM 输入：

- 用户问题
- 标的
- 时间窗口
- intent route
- 研究模型选择结果
- verified facts
- missing facts
- sources
- 用户偏好
- 禁止输出的 unsupported claims
- 报告格式要求

当前我们还没有一个清晰的：

```python
ResearchContext
```

也没有：

```python
build_research_context(run)
```

所以这段是明确缺失。

**8. 调 LLM 接口：部分具备，但没有接入最终报告**

已有：

- `AnthropicJSONResearchSynthesizer`
- `MockLLMResearchSynthesizer`
- structured synthesis schema

但当前问题：

- 默认还是 `mock`
- LLM synthesis 产物不是最终研究报告
- LLM 没有吃到完整 verified context
- LLM 没有被 claim verifier 严格约束
- renderer 现在主要是 deterministic，不是“LLM 写报告 + 事实校验”

缺失：

- evidence-constrained prompt
- context builder
- LLM 输出 schema：
  ```text
  headline_answer
  primary_causes
  secondary_causes
  ruled_out_causes
  missing_evidence
  confidence
  ```
- LLM 输出后的 unsupported claim check
- 如果 claim unsupported，自动降级或删除

**9. 用户可读研究报告：部分具备**

已有：

- 默认报告不再泄露 fact_id/source_id
- 有：
  - 研究结论
  - 原因排序
  - 发生了什么
  - 关键依据
  - 风险与不确定性
  - 还需要确认
  - 数据来源与时效

缺失：

- 真正研究员式表达仍不稳定
- 原因排序目前主要靠规则关键词
- 不能稳定生成像 ChatGPT 那样的自然分析段落
- 还缺少“主因 / 次因 / 已排除 / 置信度”更正式的结构化输出
- 缺少引用来源标题、链接、发布时间的漂亮呈现

**10. 多轮交互：基本缺失**

现在 CLI 是多轮输入，但不是多轮记忆式研究会话。

缺失：

- 上一轮问题上下文
- 上一轮标的复用
- 用户说“那它呢？”的指代解析
- 本轮 facts 和上一轮 facts 的复用
- `/sources`
- `/facts`
- `/mode live`
- `/mode fixture`

**下一步最应该补的 4 个东西**

如果要真正接近你说的目标，我建议下一步按这个顺序：

1. **Context Builder**
   把 verified facts、missing facts、time window、attribution plan、sources 组织成 LLM context。

2. **Research Model Router**
   明确选择：
   - 跌因归因模型
   - 估值模型
   - 财报模型
   - 持仓风险模型

3. **Attribution Data Executor**
   真正执行：
   - sector_move
   - peer_moves
   - index_move
   - event day price move

4. **Claim Verifier**
   LLM 写完报告后逐条检查：
   - 有证据 -> 保留
   - 没证据 -> 降级为假设或删除
   - 缺证据 -> 写入“尚不能确认”

当前离完整目标最近、也最该补的是：

```text
Context Builder + Attribution Data Executor
```

因为没有它，LLM 没有正确上下文；没有 sector/peer 数据，归因模型也只能停留在“计划和缺口”。
